🔴 Tier 1 — Fix today (monetary & data breach risk) 5 criticals
Payment security

Add auth() to /api/payments/create-order

Zero authentication on the payment creation endpoint. Anyone who knows or guesses an order UUID can initiate a Razorpay payment against it and overwrite the razorpay_order_id column. This is direct revenue manipulation.

route.ts (payments/create-order) Revenue impact

Kill the hardcoded ORDER_TOKEN_SECRET fallback

The string 'fallback-secret-key-change-in-production' is used if ORDER_TOKEN_SECRET is unset. Anyone who reads the source code can forge order tokens and download any order PDF without a QR code.

lib/utils/order-token.ts Token forgery
Unguarded server actions

Add auth + admin role gate to all coupon mutations

deleteCoupon, createCoupon, updateCoupon, toggleCouponStatus — zero auth checks. Any authenticated user (or in some Next.js configs, unauthenticated) can create unlimited 100%-off coupons or delete all active coupons.

app/actions/coupons.ts Revenue bypass

Add auth + admin role gate to inventory mutations

adjustInventoryQuantity and updateInventorySettings have no auth. Any user can set any product's stock to 0, causing a complete inventory denial-of-service — no products show as available.

app/actions/inventory.ts Stock manipulation

Fix address IDOR — add ownership to update/delete where clause

updateCustomerAddress and deleteCustomerAddress mutate by addressId alone with no ownership check. Any authenticated user knowing another's address UUID (guessable via enumeration) can delete or overwrite their shipping address.

🟠 Tier 2 — This sprint (high severity · data & performance) 5 highs
Performance bomb

Fix getOrderForTrackingAction full table scan

When a short order ID (≤8 chars) is entered, ALL orders + ALL customer_phone_snapshot values are loaded into Node memory and filtered in JS. At 10k orders this crashes the server and leaks all phone numbers to the process heap.

lib/actions/orders.ts Memory exhaustion PII leak
Auth & authorization gaps

Make getOrderForConfirmation derive userId from auth() internally

The function accepts userIdParam from the caller — if any call site passes this from a URL search param instead of auth(), it's a full IDOR. Remove the parameter, always call auth() inside the function.

lib/actions/orders.ts IDOR risk

Harden createOrder — validate userId against session

createOrder accepts userId and customerId as caller-supplied input. An authenticated user can pass another user's userId to create orders attributed to that user. If used for guest checkout, validate: if session exists, it must match the provided userId.

lib/actions/orders.ts Order spoofing

Add auth check to /api/payments/verify

Signature verification is cryptographically sound (Razorpay HMAC), but no session check means any anonymous caller with a valid triple (order_id, payment_id, signature) can mark any order as paid without being the customer.

route.ts (payments/verify) Payment spoofing

Make rate limiter fail-closed, not fail-open

When the DB is under stress, the rate limiter catches the error and returns { allowed: true } — the opposite of what a rate limiter should do under failure. During a DB degradation event, admin invite spam is unlimited.

lib/services/rate-limit.ts DoS amplifier

Environment & configuration

Build a complete startup env validator with Zod

lib/env.ts currently validates 2 of 8 required vars and falls back to empty strings. DATABASE_URL, CLERK_SECRET_KEY, ORDER_TOKEN_SECRET, RAZORPAY_WEBHOOK_SECRET, ADMIN_SETUP_KEY, ADMIN_EMAILS are all unvalidated. A missing var is only discovered at the first failed query in production.

lib/env.ts Silent misconfiguration
Security headers

Add Content-Security-Policy, HSTS, and Permissions-Policy headers

next.config.js has 4 headers but is missing the 3 most important ones. No CSP means any injected script runs freely. No HSTS means downgrade attacks are possible. No Permissions-Policy means camera/microphone APIs are unrestricted.

next.config.js XSS · MITM
Admin access control

Check is_active at middleware level for revoked admins

A revoked admin whose Clerk session is still valid can access the admin dashboard for up to 1 hour after revocation. The middleware only checks if the route requires auth — not if the admin profile is still active.

src/middleware.ts Privilege escalation

Replace ADMIN_EMAILS env gate with DB-only auth

The isAdmin() email-based gate is a secondary check that silently returns false if ADMIN_EMAILS is unset. The DB-based getAdminProfile() is already the correct primary check — the env-based gate adds confusion and an information disclosure risk (who your admins are).

Fix N+1 in getCustomersForAdmin and getCustomerStatsForAdmin

getCustomersForAdmin runs 1 query per customer (20 customers = 21 queries per page load). getCustomerStatsForAdmin loads ALL customers and ALL orders into memory. Both need SQL aggregation — GROUP BY with COUNT/SUM at the DB level.

🔵 Tier 4 — This quarter (type safety · validation · architecture)
Type safety overhaul

Enable TypeScript strict mode incrementally

strict: false is the root of 50+ as any casts. Enable noImplicitAny and strictNullChecks first (these catch 90% of real bugs), then enable strict: true once the DB TEXT/UUID mismatch is fixed and Prisma is regenerated. This is a 2-week effort but removes an entire class of runtime errors.

tsconfig.json

Add Zod validation to all API route bodies

All request bodies are consumed as TypeScript casts only — no runtime validation. A caller sending { orderId: null, amount: "abc" } proceeds silently. Add Zod schemas to every POST/PATCH route — this also auto-generates type-safe inputs, replacing as any patterns.

All route.ts files
Architecture upgrades

Implement structured logging with Pino

console.log in payment flows leaks PII and has no log levels, correlation IDs, or structured fields. Replace with Pino — log structured JSON with request ID, user ID (not email), severity level. This also integrates with most log aggregation platforms cleanly.

Add timingSafeEqual to verifyRecoveryKey

Recovery key comparison uses === string equality — timing attacks are theoretical on 32-char random keys, but crypto.timingSafeEqual() is the correct pattern and costs zero effort to implement.

lib/admin-auth.ts

Add SSL config explicitly to the DB pool

The pg.Pool does not set ssl: { rejectUnauthorized: true } — it relies on sslmode=require being in the DATABASE_URL string. If the connection string is modified without the SSL param, connections silently downgrade to plaintext. Enforce SSL in code, not just config.

lib/db.ts

Create a reusable withAdminAuth server action wrapper

After fixing the 3 unguarded server actions, you will have repeated auth+role-check boilerplate in every admin action. Extract this into a higher-order function so future actions cannot accidentally omit the guard — it becomes impossible to write an unprotected admin action.
