🟠 Sprint 2 — DB migration (2–3 hrs · run on dev branch first) 1 item · ~3 hrs

Deprecate orders.user_id — migrate fully to customer_id

The dual-identity problem: orders has both user_id (Clerk TEXT string) and customer_id (UUID FK). RLS currently covers both with an OR clause — this adds query overhead on every RLS check and creates confusion about which field is authoritative. Migrate all remaining references from user_id to customer_id, rename the column to _deprecated_user_id, verify for 2 weeks, then drop it. This also unblocks TypeScript strict mode since the dual-field OR pattern is the last source of type ambiguity in order queries.

orders table · lib/actions/orders.ts · middleware.ts Run on Neon dev branch Unblocks TS strict mode

🏗️ Sprint 3 — Architecture additions (enterprise-grade) 4 items · ~7 hrs
Observability & compliance

Implement audit_log table + triggers

No trail exists for who changed what on orders, customers, or products. Needed for fraud investigation, customer disputes, and GDPR right-to-access. A single audit_log table with BEFORE/AFTER JSONB snapshots on critical tables covers all three use cases.

neondb (new table) · DB triggers GDPR compliance Fraud investigation

Add soft delete pattern (deleted_at) to critical tables

Hard DELETEs on orders and customers make GDPR erasure requests dangerous — you can't anonymise what's already gone, and you can't prove it was erased. Add deleted_at nullable timestamp + update RLS to exclude soft-deleted rows + a separate hard_purge() function for actual GDPR erasure.

orders · customers · products GDPR erasure path
2 hrs

Replace console.log with Pino structured logging

Payment flows still use console.log with redacted PII (good), but there's no log levels, request correlation IDs, or structured JSON format. Pino integrates with Vercel/Hostinger log drains and makes debugging production issues possible without guessing from raw stdout.

Fix middleware error handler — redirect to login, not pass-through

If middleware throws unexpectedly, NextResponse.next() is returned — the user passes through unauthenticated. Low probability but the consequences are high (unauthenticated access to protected routes). A redirect to /login or a 500 is the correct failure mode.

🔒 Sprint 4 — CSP nonce hardening (optional · complex) 1 item · 4–6 hrs

Replace unsafe-inline + unsafe-eval with nonce-based CSP

Current CSP includes unsafe-inline and unsafe-eval in script-src — required by Razorpay checkout and Next.js hydration. These directives weaken the XSS protection the CSP provides. A nonce-based approach generates a per-request cryptographic nonce, threads it through Next.js middleware and the Razorpay script tag — completely removes unsafe-inline. Complex to set up correctly with Clerk and Razorpay but is the gold standard for CSP.
