🔴 Tier 1 — Immediate (security · do today) Blocks production readiness
Row-level security

Enable RLS on all user-owned tables

customers, orders, addresses, carts, wishlists — any authenticated DB connection currently reads every user's data. This is a full data breach risk in production.

GDPR / data breach ~2 hrs work

Protect neon_auth session & account tables

OAuth tokens and session secrets are currently readable by any authenticated role. Enable RLS with userId = auth.uid() policy on both tables.

Token hijack risk

Add separate ADMIN bypass policy

After RLS is on, admin dashboard queries will fail unless you create a BYPASSRLS role or an explicit admin policy. Do this before enabling RLS to avoid outage.

Do before RLS goes live

This TEXT/UUID mismatch prevents adding a real FK constraint on orders → customers. Run the migration on a Neon dev branch first, then promote.

Foreign keys & orphans

Add 5 missing FK constraints

inventory_items.variant_id, cart_items.user_id, wishlists.user_id, product_analytics.product_id, addresses.user_id all reference parent tables without formal FK constraints — orphan risk over time.

4 wishlist rows reference non-existent user_ids. Delete or nullify before adding the FK constraint or it will fail.

Run on branch first
Index hygiene

Drop 4 confirmed zero-scan indexes

idx_orders_customer_id (136 kB), idx_product_collections_collection_id, idx_product_variants_product_id, idx_addresses_user_id — all 0 scans, pure write overhead. Use DROP INDEX CONCURRENTLY to avoid table lock.

Add composite index on orders(customer_id, status)

The orders table has 1,750 rows and high write activity. A composite index covering the most common query pattern (fetch orders by customer + filter by status) will pay off immediately and replace the dropped single-column index.

Schema cleanup

Deprecate addresses.user_id column

The audit found user_id is deprecated in addresses in favour of customer_id. Rename to _deprecated_user_id, verify no app code reads it for 2 weeks, then drop.

Add updated_at to all tables missing it

Industry standard — every mutable table needs created_at + updated_at. Add the column + a BEFORE UPDATE trigger to auto-stamp. Needed for cache invalidation, audit logs, and sync.

🔵 Tier 3 — This sprint (architecture upgrades)
Audit & compliance

Implement audit log table

Enterprise standard — a single audit_log table recording who changed what and when across orders, customers, products. Implement via trigger or application layer. Needed for fraud investigation and GDPR right-to-access requests.

Add soft delete pattern (deleted_at)

Replace hard DELETEs on orders, customers, products with a deleted_at nullable timestamp. Adds GDPR-safe anonymisation path and prevents accidental permanent data loss.

Performance at scale

Partition orders table by created_at

1,750 rows now, but orders tables grow unboundedly. Partition by month (RANGE on created_at) before it hits 100k rows. Far cheaper to set up now than to migrate later.

Enable pg_cron for auto-vacuum scheduling

system_preferences (300% bloat), products (119%), product_collections (188%) bloat fast. Schedule VACUUM ANALYZE weekly via pg_cron so autovacuum lag never accumulates again.

Introduce DB connection pooling (PgBouncer)

Neon supports PgBouncer pooling. E-commerce apps spike on flash sales — without pooling, 100 concurrent checkouts = 100 PG connections. Enable transaction-mode pooling and update connection strings.

Neon-specific

Formalise dev → staging → production branch workflow

Use Neon's branching to run every migration on a dev branch first, promote after QA. Lock the production branch so direct DDL is blocked — all schema changes must come through CI.

Set up Point-in-Time Recovery (PITR) checkpoints

Neon's time-travel lets you restore to any point. Document recovery runbooks and test restoring to 1hr ago on a dev branch quarterly — so the first time you need it isn't also the first time you've tried it.

Observability

Enable pg_stat_statements + query monitoring

The audit had limited query-level data because pg_stat_statements wasn't active. Enable it to catch N+1 queries, slow reports, and index misses before users feel them.
